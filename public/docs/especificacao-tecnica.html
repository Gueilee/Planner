Especificação Técnica — v1.0.0 1. Visão Geral Técnica
Seção 01
Visão Geral Técnica
Esta seção apresenta o objetivo técnico do sistema Planner, a visão arquitetural de alto nível, as estratégias de escalabilidade e as diretrizes gerais de segurança que orientam todas as decisões de implementação.

1.1 Objetivo Técnico

O Planner é uma aplicação web full-stack moderna, construída sobre o paradigma de Server-Side Rendering (SSR) híbrido com hidratação no cliente, que provê uma plataforma corporativa de gestão de projetos. Tecnicamente, o sistema foi projetado para:

  • Entregar interfaces de alta fidelidade com tempo de carregamento inicial abaixo de 1,5 s (LCP) utilizando renderização no servidor via Next.js App Router com React Server Components (RSC)
  • Persistir dados relacionais com integridade referencial por meio de Prisma ORM sobre banco de dados SQLite distribuído (libsql/Turso), com migração transparente para PostgreSQL em ambientes de alta concorrência
  • Garantir autenticação segura baseada em sessões JWT assinadas com rotação automática via NextAuth v5, com suporte a múltiplos provedores de identidade
  • Gerenciar uploads de arquivos e evidências de forma segura e escalável por meio do Vercel Blob (CDN-backed object storage), com URLs assinadas e controle de acesso por projeto
  • Suportar múltiplas organizações por meio de configuração singleton (OrgConfig), permitindo parametrização de identidade visual, logotipo e nomenclaturas

1.3 Arquitetura Geral — Visão de Alto Nível

O sistema segue o padrão Monolito Modular (modular monolith) dentro do Next.js App Router, onde cada domínio funcional possui seu próprio conjunto de rotas, componentes, stores e tipos. Esta abordagem permite escalabilidade de equipe sem os custos operacionais de microsserviços prematuros.

Camada de Apresentação (Client Layer)

React 19 com Concurrent Features, Server Components para renderização estática e Client Components para interatividade. Tailwind CSS 4 com design system baseado em tokens. Zustand para estado global, TanStack Query para cache e sincronização de dados servidor-cliente.

Camada de Aplicação (App Layer — Next.js)

App Router com layouts aninhados, grupos de rotas por contexto (dashboard), (print), (auth). Route Handlers para APIs REST internas. Middleware de autenticação para proteção de rotas. Server Actions para mutações diretas sem API explícita.

Camada de Dados (Data Layer — Prisma + libsql)

Prisma ORM como abstração de banco de dados com 16 modelos relacionais. libsql com Turso para SQLite distribuído com replicação multi-região. Migrações versionadas e gerenciadas pelo Prisma Migrate. Seed de dados via script TypeScript.

Camada de Armazenamento (Storage Layer — Vercel Blob)

Vercel Blob para storage de objetos (arquivos, imagens, PDFs gerados). URLs públicas com CDN global. Integração nativa com o ecossistema Vercel. Sem limite prático de tamanho de arquivo configurado.

Camada de Identidade (Auth Layer — NextAuth v5)

NextAuth v5 com Prisma Adapter para persistência de sessões. JWT assinado com segredo NEXTAUTH_SECRET. Hash de senhas com bcryptjs (salt rounds: 12). RBAC implementado via campo UserRole no modelo User com 6 papéis distintos.

1.5 Princípios Técnicos Adotados

  • Type Safety End-to-End: TypeScript em toda a stack com inferência de tipos Prisma gerados automaticamente, eliminando classes de erros em runtime
  • Server-First: Dados sensíveis processados no servidor via RSC e Route Handlers; cliente apenas exibe e interage
  • Zero-Secrets-on-Client: Nenhuma credencial, chave de API ou segredo exposto ao bundle do cliente
  • Validação em Duas Camadas: Zod no cliente (UX imediato) e Zod no servidor (segurança real), nunca confiando apenas no cliente
  • Otimismo Controlado: TanStack Query com optimistic updates para percepção de velocidade; rollback automático em caso de erro do servidor
  • Convenção sobre Configuração: Estrutura de pastas Next.js App Router seguida rigorosamente, reduzindo curva de onboarding de novos desenvolvedores

2.2 Padrão de Comunicação entre Camadas

1
Browser → Next.js Server

O usuário interage com Client Components que disparam mutations via TanStack Query (useMutation) para as Route Handlers da API interna, ou via Server Actions para operações diretas no servidor.

↓
2
Middleware de Autenticação

O arquivo middleware.ts intercede em cada request, verificando a sessão JWT via NextAuth. Rotas protegidas redirecionam para /login se não autenticado.

↓
3
Route Handler → Prisma → libsql

As Route Handlers utilizam o cliente singleton do Prisma (lib/db.ts) para executar queries type-safe. O adapter libsql traduz as queries Prisma para SQL SQLite executado no Turso.

↓
4
Resposta e Cache

O Route Handler retorna NextResponse.json(). No cliente, o TanStack Query armazena em cache e invalida automaticamente as queries relacionadas após mutations bem-sucedidas.

↓
5
Reatividade

Zustand stores mantêm estado local (modais abertos, seleção de itens, filtros) sem necessidade de prop drilling. Componentes se inscrevem apenas nos slices do estado que necessitam, evitando re-renders desnecessários.

2.5 Fluxo de Autenticação

O sistema usa NextAuth v5 com o provider Credentials, suportando login com e-mail e senha. O fluxo completo:

  1. Usuário submete credenciais no formulário /login
  2. NextAuth executa o callback authorize(): busca usuário no banco via Prisma, compara senha com bcrypt
  3. Se válido, cria sessão JWT contendo { id, email, name, role, image } assinado com NEXTAUTH_SECRET
  4. Cookie authjs.session-token armazenado como HttpOnly; Secure; SameSite=Lax
  5. Middleware valida o JWT em cada request subsequente sem consultar o banco de dados
  6. Server Components acessam sessão via await auth(); Client Components via useSession()
Observação de Arquitetura

A abordagem Credentials Provider (email+senha) é a mais simples e foi escolhida para o contexto corporativo fechado onde o administrador gerencia os usuários manualmente. Para expansão futura com SSO corporativo (Microsoft Entra ID / Google Workspace), NextAuth v5 suporta adição de OAuth Providers sem refatoração da arquitetura de sessão.

2.6 Geração de Documentos PDF

A geração de ATAs e relatórios em PDF utiliza uma abordagem HTML-to-Print sem dependências de bibliotecas pesadas de PDF no servidor. O fluxo é:

  1. O sistema monta um documento HTML completo (com estilos inline e dados do projeto) dentro de um grupo de rotas (print)
  2. O usuário acessa a rota de impressão em nova aba (window.open)
  3. O browser renderiza o documento com @media print CSS optimizado para papel A4
  4. O usuário utiliza Ctrl+P → Salvar como PDF com configurações pre-definidas (sem cabeçalho/rodapé do browser)
  5. Opcionalmente, o sistema pode armazenar o PDF gerado no Vercel Blob para arquivamento
Vantagem da Abordagem HTML-to-Print

Sem dependência de puppeteer, wkhtmltopdf ou bibliotecas de PDF no servidor (que consomem memória e adicionam cold start). O browser nativo gera PDFs de alta qualidade com suporte completo a fontes, gradientes e layout A4. Esta abordagem é compatível com o ambiente serverless da Vercel.

Limitação Atual — SQLite

O SQLite (via libsql/Turso) não suporta concorrência de escrita nativa como PostgreSQL. Em ambientes com alto volume de escritas simultâneas (>100 req/s de mutação), recomendar migração para PostgreSQL via troca do datasource no schema Prisma e adição do adapter @prisma/adapter-neon ou @prisma/adapter-pg. O Prisma Schema permanece 100% compatível.

4.6 Rate Limiting

Na infraestrutura Vercel atual, o rate limiting é gerenciado pela plataforma (límites por plano). Para ambientes de alto volume ou migração futura, a estratégia recomendada é:

  • Implementação de middleware de rate limiting baseado em IP usando @upstash/ratelimit com Redis (Upstash)
  • Limites recomendados: 100 req/min por IP para endpoints públicos, 300 req/min por usuário autenticado
  • Headers de resposta: X-RateLimit-Limit, X-RateLimit-Remaining, Retry-After
  • Resposta 429 Too Many Requests com body { error: "Rate limit exceeded" }

5.4 Criptografia e Proteção de Dados

  • Em Trânsito: HTTPS/TLS 1.3 obrigatório — fornecido automaticamente pelo Vercel em todos os deployments. Certificados Let's Encrypt gerenciados pela plataforma
  • Em Repouso (Senhas): bcrypt com salt único por usuário, cost factor 12 (~300ms). Senhas nunca armazenadas em texto claro ou reversíveis
  • Em Repouso (Banco): Turso (libsql) criptografa dados em repouso por padrão no plano cloud (AES-256)
  • Secrets: Gerenciados via Vercel Environment Variables (encrypted at rest, never in source code)
  • JWT Secret: NEXTAUTH_SECRET — string de 64 caracteres gerada via openssl rand -base64 48

Cache no Cliente — TanStack Query
  • staleTime: Dados considerados frescos por 30 segundos — sem re-fetch desnecessário em navegações rápidas
  • gcTime: Cache mantido por 5 minutos após o componente desmontar — dados disponíveis ao voltar para a página

  • Dados buscados em Server Components usam o cache do Next.js com { next: { revalidate: 60 } } para dados semi-estáticos
  • Dados dinâmicos (projetos em edição) usam { cache: 'no-store' } para garantir freshness
  • OrgConfig (nome/logo da organização) cacheado por 3600s — muda raramente

8.3 Code Splitting e Lazy Loading

  • Rota-baseado (automático): Next.js cria chunks separados por rota. Usuário baixa apenas o código da página atual
  • Dynamic Imports: Componentes pesados (editor Tiptap, charts Recharts, DnD Kit) carregados sob demanda com next/dynamic
  • Suspense Boundaries: Loading states explícitos com <Suspense fallback={<Skeleton />}> para cada seção da UI
  • Imagens: Componente next/image para otimização automática (WebP, lazy loading, blur placeholder)

8.4 Otimizações de Banco de Dados

  • Select explícito: Queries Prisma com select: {} retornam apenas campos necessários, reduzindo payload e tempo de deserialização
  • Include seletivo: Relations carregadas apenas quando necessárias para a view — evita N+1 queries
  • Índices estratégicos: Colunas usadas em WHERE, ORDER BY e JOIN têm índices definidos no schema
  • Soft Deletes: Campo active em User evita hard-delete com cascata; queries filtram active: true
  • Batch Operations: Múltiplas criações (ex: importar cronograma de Excel) usam createMany() em única transação

O pipeline é configurado via integração nativa entre GitHub e Vercel. Cada evento no repositório dispara ações automáticas:

Push em feature/* ou fix/* → Preview Deployment

1. Vercel detecta o push via GitHub webhook. 2. Executa o build (prisma generate && next build) com as env vars do ambiente Preview. 3. Deploy em URL única (ex: kronex-abc123.vercel.app). 4. Vercel comenta no PR com o link do Preview. 5. Equipe valida a feature no Preview antes de aprovar o merge.

Pull Request → Code Review Obrigatório

Checklist de PR: TypeScript sem erros (tsc --noEmit), ESLint sem warnings bloqueantes, Preview Deployment aprovado, ao menos 1 aprovação de revisor. Merge squash para manter histórico limpo no main.

Merge em main → Deploy Automático em Produção

1. Build em produção com env vars de PRD. 2. Migrações pendentes do Prisma executadas via script de migração (separado do deploy — executado manualmente antes do merge para evitar downtime). 3. Deploy atômico na Edge Network. 4. Health check automático. 5. Notificação no Slack/Teams com link do deployment.

10.4 Gestão de Migrações de Banco

1
Alterar schema.prisma

↓
2
npx prisma migrate dev --name descricao

↓
3
Commit da migração junto ao código

O arquivo de migração é versionado no Git junto com o código que o requer. Nunca separar migração de código.

↓
4
npx prisma migrate deploy (pré-deploy)

Antes do merge em main, executar migrate deploy no banco de staging e produção. Este comando aplica apenas migrações pendentes, de forma idempotente.

Regra de Ouro — Migrações Seguras

Nunca remover ou renomear colunas diretamente em uma migração de deploy. Sempre seguir o padrão de 3 passos: (1) adicionar nova coluna nullable, (2) migrar dados, (3) remover coluna antiga em release separado. Isso garante zero downtime e rollback seguro.

12.2 Migração de Banco — SQLite para PostgreSQL

A migração de libsql para PostgreSQL requer apenas 3 alterações no codebase, sem impacto nos modelos ou queries Prisma:

schema.prisma — antes e depois
// ANTES (atual)
datasource db {
  provider = "sqlite"
  url      = env("TURSO_DATABASE_URL")
}

// DEPOIS (PostgreSQL — Neon ou Supabase)
datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")
  directUrl = env("DIRECT_URL")  ← para migrations
}

O adapter libsql em lib/db.ts é removido e substituído pelo Prisma Client padrão. Todos os 16 modelos, relações e queries permanecem idênticos. Tempo estimado de migração: 1 dia de trabalho + janela de migração de dados.

12.4 API Pública — Estratégia de Extensão

Para suportar integrações com sistemas externos (ERP, Power BI, ferramentas de BI), a evolução natural é expor uma API REST pública documentada:

12.5 Multi-tenancy — Isolamento por Organização

Para suportar múltiplas organizações no mesmo deployment (modelo SaaS), a estratégia recomendada é Row-Level Security (RLS) com campo orgId em todos os modelos:

  • Adicionar modelo Organization e campo orgId em User, Project e demais modelos
  • Middleware Prisma que injeta automaticamente where: { orgId: session.user.orgId } em todas as queries
  • Subdomínio por organização (vendemmia.planner.com.br) resolvido via middleware Next.js
  • Isolamento de dados garantido em nível de ORM — sem possibilidade de vazamento entre organizações

12.7 Considerações Finais de Arquitetura

Princípio Orientador de Evolução

O Planner foi construído com a filosofia de "simplicidade agora, extensibilidade depois". A escolha do Next.js monolito modular permite que o time entregue valor rapidamente sem overhead de microsserviços. Quando um domínio crescer suficientemente (ex: Analytics, Notificações, API Pública), ele pode ser extraído para um serviço dedicado sem refatorar o restante do sistema. O Prisma Schema é a fonte única da verdade — qualquer refatoração de infraestrutura começa e termina ali.

Resumo do Posicionamento Técnico

O Planner está implementado sobre um stack moderno, production-ready e alinhado com as melhores práticas da indústria (2025/2026). A combinação Next.js + Prisma + Vercel oferece velocidade de desenvolvimento 3-5× superior a stacks tradicionais (Java/Spring, .NET MVC), com escalabilidade suficiente para suportar centenas de usuários simultâneos sem custo de operações (DevOps). A arquitetura está preparada para crescer junto com as necessidades da Vendemmia.
